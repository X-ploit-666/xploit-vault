> 01_CLIENT_SIDE_DANGERS

Everything we have talked about—HTML, CSS, and JavaScript—happens on the Client-Side. This means the code runs directly on the user's computer, not on the company's main hidden server (the Back End). Because of this, an attack on these parts usually doesn't break the main server forever. However, it puts the User in great danger.

If a hacker finds a weakness in these front-end parts, they can target important people, like the website administrators. If they trick an admin, they can steal secret data, lock people out of the site, or even find a way to jump from the front end into the main back-end server to take full control.

> SENSITIVE_DATA_EXPOSURE

Sensitive Data Exposure happens when a website accidentally leaves private information in plain view for anyone to see. This usually happens in the Source Code—the raw HTML and JavaScript that makes up the page. You can see this code on almost any website by right-clicking and choosing "View Page Source" or pressing Ctrl + U.

In this hidden code, hackers often find "low-hanging fruit"—easy mistakes that developers made. This includes forgotten usernames and passwords, secret links to hidden parts of the site, or private details about the users. One of the first things a security expert does is scan this source code to see what the developers accidentally left behind.

> 02_PREVENTION_STRATEGY

To keep a website safe, the front-end code should be "clean." It should only contain the bare minimum code needed to run the site. Developers should never leave notes (comments) about passwords or hidden links. They should also use a trick called Obfuscation (making the code look like a messy, unreadable jumble) to make it much harder for hackers or automated tools to find any secrets.

> 03_VULNERABILITY_ANALYSIS

This code looks like a normal login form on the outside, but the source code tells a different story.

<form action="action_page.php" method="post">
    <div class="container">
        <label for="uname"><b>Username</b></label>
        <input type="text" required>

        <label for="psw"><b>Password</b></label>
        <input type="password" required>

        <!-- TODO: remove test credentials test:test -->

        <button type="submit">Login</button>
    </div>
</form>
Part of Code What it actually means / The Danger
<form action="..."> This tells the browser where to send the username and password when you click login. Here, it sends it to a file called action_page.php.
type="password" This is why you see dots (****) instead of letters when you type your password. It hides the text from anyone standing behind you.
<!-- ... --> This is an HTML Comment. The browser ignores this text and does not show it on the webpage. It is meant only for the developers to read.
TODO: remove test credentials test:test CRITICAL LEAK: The developer left a note to themselves. The text test:test is likely a valid username and password for testing the site. A hacker can now log in without permission.

> 04_THE_ATTACK_PATH

How "Front-End" leaks lead to "Back-End" disasters:

[ THE HACKER ] | v [ VIEW PAGE SOURCE ] (Ctrl + U) | +-- Finds [ COMMENT ] (e.g., "admin:pass123") | v [ LOGIN AS ADMIN ] (Authorized Access) | +-- Accesses [ SENSITIVE DATA ] (Emails, Credit Cards) | +-- Accesses [ SERVER SETTINGS ] | v [ TOTAL SERVER COMPROMISE ] <--- THE GOAL

Security Best Practices Checklist:

+---------------------------------------------------+ | [ ] REMOVE ALL DEVELOPER COMMENTS | | [ ] DELETE "TEST" ACCOUNTS AND HIDDEN LINKS | | [ ] USE JAVASCRIPT OBFUSCATION (SCRAMBLING) | | [ ] SCAN CODE WITH AUTOMATED SECURITY TOOLS | | [ ] NEVER TRUST THE CLIENT-SIDE FOR SECURITY | +---------------------------------------------------+
