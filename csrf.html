> 01_THE_UNWANTED_REQUEST

Cross-Site Request Forgery (CSRF) is a trick where a hacker makes your browser perform an action on a website without you knowing. Imagine you are logged into your bank. You visit a hacker's "malicious" site in another tab. That site contains a hidden command that tells your browser: "Tell the bank to send $1000 to the hacker." Because you are already logged in, the bank thinks you sent the command!

This attack often uses XSS to work. A hacker might post a comment on a forum that contains a hidden script. When you read that comment, the script automatically sends a request to change your password to something the hacker knows. Suddenly, you are locked out, and the hacker is in.

> THE_ADMIN_TRAP

Hackers love targeting administrators with CSRF. Since admins have the "keys to the kingdom," tricking an admin's browser into deleting a user, changing a site setting, or granting the hacker admin powers can lead to a total server takeover.

> 02_HOW_THEY_ATTACK

The exploit.js file is a custom-made tool that knows exactly how the target website works. It automatically fills out forms (like the "Change Password" form) and clicks "Submit" in the background before you even realize anything happened.

> 03_THE_DEFENSE_PROTOCOL

To stop these attacks, developers must build "Security by Design." There are two main ways to handle user input:

Control The Hacker-Proof Explanation
Sanitization Cleaning the input. If a user types <script>, the cleaner turns it into plain text so it can't run.
Validation Checking the format. If a box asks for an Email, the site should reject anything that doesn't look like user@example.com.

> MODERN_SHIELDS

  • Anti-CSRF Tokens: A secret, one-time code added to every request. If the code is missing or wrong, the server rejects the request.
  • SameSite Cookies: A setting that tells the browser: "Don't send my login session to other websites."
  • Password Confirmation: Forcing users to type their old password before setting a new one stops CSRF dead in its tracks.
