### **TOPIC: BOUNDARY VALIDATION STRATEGIES**

#### **SECTION 1 — Simplified Version** A common way to stop Cross-Site Scripting (XSS) is to change dangerous characters into a safe HTML format before they are put onto a webpage. However, cleaning data (sanitization) is hard if one piece of user input contains many different types of potentially bad data. In these cases, it is better to use "boundary validation."

Many security holes happen because user data is handled dangerously. You can avoid these by making the way you process data safe by default, rather than just checking the input. For example, "parameterized queries" stop SQL Injection by design. You can also design the app to avoid dangerous habits, like sending user input directly to the computer’s command center (the OS shell). While this "safe-by-design" method doesn't work for everything, it is very effective.

#### **SECTION 2 — Explanation**

In cybersecurity, we assume all user input is "poisoned" until proven otherwise.

- **Why it matters:** If an app just tries to "clean" a list of bad words, hackers will find a way around it.
    
- **The Shift:** Instead of just playing "cat and mouse" with bad input, we use **Safe Data Handling**. This means using coding tools that treat data as _just data_, never as _executable code_. This is the difference between trying to take a bomb apart (sanitization) and putting the bomb in a reinforced box where it can't hurt anyone (safe processing).
    

**Technical Terms:**

- **Cross-Site Scripting (XSS):** An attack where a hacker injects malicious scripts into a webpage viewed by other users.
    
- **HTML-encode:** Converting special characters like `<` into `&lt;` so the browser displays them as text instead of running them as code.
    
- **Sanitization:** The process of cleaning or filtering user input to prevent it from causing harm.
    
- **SQL Injection:** An attack where a hacker inserts database commands into an input field to steal or delete data.
    
- **Parameterized Queries:** A way of talking to a database where the "command" and the "user data" are kept completely separate so the data can't be mistaken for a command.
    

#### **SECTION 3 — Key Points to Remember**

- 🟢 **Sanitization is difficult** when input serves multiple purposes.
    
- 🟢 **Safe Programming** (like parameterized queries) is better than just "checking" input.
    
- 🟢 **Design Choice:** Avoid passing user data directly to the Operating System.
    
- 🟢 **Boundary Validation:** Check data at every point where it moves from one part of the system to another.

### **TOPIC: MULTISTEP VALIDATION & CANONICALIZATION**

#### **SECTION 1 — Simplified Version**

A big problem happens when an app changes or cleans user input in multiple steps. If not done perfectly, a hacker can hide bad data that only "appears" after the cleaning is done. For example, if an app deletes the word `<script>` from input, a hacker might send `<scr<script>ipt>`. When the app deletes the middle "script" tag, the leftover letters collapse together to form a new, dangerous `<script>` tag. This happens because the filter didn't check the data a second time (recursively).

This also happens with "canonicalization" (converting data into its simplest form). Browsers encode data so it can travel over the web safely. If an app filters data _before_ it decodes it, a hacker can use double-encoding (like `%2527`) to sneak past. The filter sees safe text, but after the app decodes it, it turns into a dangerous apostrophe (`'`).

#### **SECTION 2 — Explanation**

Hackers love "logic flaws" in how data is transformed.

- **Why it matters:** Security filters are often like a gate that only checks you once. If you can wear a "costume" (encoding) that the gate doesn't recognize, but take it off once you're inside, you've bypassed the security.
    
- **The Danger:** If validation steps are in the wrong order, or if the app "best-fits" characters (like changing `«` to `<`), the security check becomes useless because it's looking at the wrong version of the data.
    

**Technical Terms:**

- **Recursive:** A process that repeats itself until the job is completely done (e.g., checking for bad words again after deleting some).
    
- **Canonicalization:** The process of turning different versions of data (like encoded text) into one standard, "normal" format.
    
- **URL Encoding:** A way of translating special characters into a `%` followed by numbers so they can be sent in a URL.
    

#### **SECTION 3 — Key Points to Remember**

- ⚠️ **Filter Bypasses:** Deleting bad words can actually create new ones (The "Collapse" effect).
    
- ⚠️ **Step Order:** If you filter _before_ decoding, the filter is blind to the hidden threat.
    
- ⚠️ **Character Mapping:** "Best fit" conversion (e.g., `»` becoming `>`) can be used to smuggle code.
    
- ⚠️ **Solution:** Reject "bad" input entirely rather than trying to fix it, or use recursive cleaning.
