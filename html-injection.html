> 01_THE_UNFILTERED_TRAP

A major part of web security is making sure that anything a user types into a box—like their name or a comment—is "cleaned" before it is shown on the page. This cleaning process is called Validating (checking if the info is correct) and Sanitizing (removing dangerous characters).

HTML Injection happens when a website takes what a user typed and puts it directly onto the page without cleaning it first. If a user types normal text, it looks fine. But if a user is a hacker, they can type HTML code instead of a name. The web browser will read that code and think it's part of the real website!

This is dangerous because a hacker can use it to "deface" a website—which means changing how the page looks, adding fake ads, or even putting a fake login box on the screen to steal passwords. This can make a company look very bad and lose the trust of their customers.

> HOW_IT_SPREADS

There are two main ways this happens. One is when the website saves your dangerous code in a database (like a comment section) and then shows it to everyone who visits later. The other way is when JavaScript on the page takes what you just typed and immediately puts it on the screen without checking it.

> 02_VULNERABLE_CODE_BREAKDOWN

This code allows a user to enter their name, but it has zero protection. Let's see why.

<button onclick="inputFunction()">Click to enter your name</button>
<p id="output"></p>

<script>
    function inputFunction() {
        // 1. Ask the user for their name
        var input = prompt("Please enter your name", "");

        if (input != null) {
            // 2. Put that input DIRECTLY into the page
            document.getElementById("output").innerHTML = "Your name is " + input;
        }
    }
</script>
Part of Code The Beginner Explanation
prompt(...) This pops up a little box on your screen asking you to type something. Whatever you type is saved in a variable called "input".
document.getElementById("output") This finds the empty paragraph on the page where the name should go.
.innerHTML = ... + input THE VULNERABILITY: This command tells the browser to treat the user's "input" as actual HTML code. If the user types a name, it works. If they type code, the browser runs it.

> 03_THE_ATTACK_PAYLOAD

In this example, a hacker doesn't type a name. They type a "Payload" (malicious code) like this:

<style> body { background-image: url('https://attack-site.com/logo.svg'); } </style>

Because the website uses .innerHTML, it takes that "Style" code and adds it to the page's brain. Instantly, the background of the whole website changes to whatever image the hacker chose. If the hacker were more dangerous, they could use this to steal information instead of just changing a background.

> 04_SECURITY_VISUALIZATION

The flow of a successful HTML Injection:

[ USER INPUT BOX ] | | "<h1>HACKED</h1>" v [ JAVASCRIPT LOGIC ] <-- NO CLEANING / NO FILTERING | | Directly injects into .innerHTML v [ THE DOM (PAGE MAP) ] | | The browser reads "<h1>" as a command v [ THE SCREEN ] <-- Shows a giant red "HACKED" title

How to Fix (Sanitization):

[ USER INPUT ] ----> [ SANITIZER FILTER ] ----> [ SAFE OUTPUT ] "<b>Hi</b>" Removes "<" and ">" "Hi" (Result: The browser sees plain text, not a command!)
