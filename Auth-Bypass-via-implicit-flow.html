The Critical Logic Gap

The server has no password to check. It trusts two things provided by the user's browser:

  • ⚠ Email: Who you claim to be
  • ⚠ Token: Proof you logged in somewhere else

If the server doesn't link the token to the specific email, identity becomes a variable we can edit.

The Club Analogy

Imagine a high-end club entry system where the guard is lazy.

THE PROCESS:

You show a valid ticket (Token) and say "I'm VIP John" (Email).

The guard checks that the ticket is real, but never checks if the ticket was issued to John.

Why this is "Evil" Design?

01. BROWSER EXPOSURE

Implicit flow puts the token in the URL. It's visible, loggable, and easily grabbed.

02. USER-CONTROLLED IDENTITY

The POST request is sent from the client. Anything the client sends can be manipulated before it hits the server.

03. BLIND BACKEND TRUST

The server assumes the browser is honest. It fails to re-verify the token-user link on its own side.

04. PASSWORDS ARE OBSOLETE

The attacker doesn't need to know the victim's password. They only need to know the victim's email.

Final Verdict

"Implicit flow is a security nightmare when servers trust user-supplied identity without verifying the token ↔ user binding."
