### What are Public Vulnerabilities?

The most dangerous weaknesses are found in the **Back-end** (the hidden "engine room" of a website). These are bad because a hacker can use them from their own home without ever having physical access to the website's computer.

These weaknesses usually happen because a programmer made a mistake while writing the code. Some are easy to fix, while others are very complicated and require a lot of expert knowledge to understand.

### 🔍 Finding Public "CVEs"

Many companies use "Open-Source" software (code that anyone can see and use). Because so many people look at this code, experts around the world find bugs all the time.

1. **CVE (Common Vulnerabilities and Exposures):** When a bug is found and fixed, it gets a special ID number called a CVE. This is like a "Name Tag" for a specific security hole.
    
2. **Exploits:** Security testers often create "Proof of Concept" code. This is a small program that proves a bug actually works. They share these online so others can learn or test their own systems.
    
3. **The First Step:** The very first thing a security tester does is find the **Version Number** of the website software (e.g., WordPress 5.4). They look for this in the website's code or special files like `version.php`.
    

Once they have the version number, they search Google or special websites like **Exploit DB** or **Rapid7** to see if there are any "ready-to-use" hacks for that specific version.

### 📈 Understanding the Score (CVSS)

How do we know which bug is the most dangerous? We use the **CVSS (Common Vulnerability Scoring System)**. It gives every bug a score from **0 to 10**.

- **0.0:** No risk.
    
- **9.0 - 10.0:** **Critical!** This is an "all hands on deck" emergency.
    

The score is calculated using three main "ingredients":

1. **Base Score:** How bad is the bug by itself? (This is the main number people look at).
    
2. **Temporal Score:** Does the bug change over time? (For example, is there a fix available yet?).
    
3. **Environmental Score:** How much damage would this bug do to _your_ specific company?
    

**V2 vs V3:** There are two versions of this scoring system. **V3** is newer and more detailed, adding a "Critical" category for the very worst bugs.

|Severity|V2 Score Range|V3 Score Range|
|---|---|---|
|**None**|N/A|0.0|
|**Low**|0.0 - 3.9|0.1 - 3.9|
|**Medium**|4.0 - 6.9|4.0 - 6.9|
|**High**|7.0 - 10.0|7.0 - 8.9|
|**Critical**|N/A|9.0 - 10.0|

### 🖥️ Server and Database Weaknesses

It's not just the website code that can be weak; the **Web Server** (the computer hosting the site) can have bugs too.

- **Remote Code Execution (RCE):** This is the "Holy Grail" for hackers. It lets them run any command they want on the server from far away. A famous example is **Shell-Shock**, which let hackers take over servers through a simple web request.
    
- **Internal Vulnerabilities:** Some bugs can't be reached from the internet. A hacker first has to get "inside" the company network using a public bug. Once they are inside, they use these internal bugs to move from one computer to another or to become the "Super User" (Administrator).
    

### 🗺️ The Path of a Security Search

**START HERE** 🏁 | V **Identify Version** 🔍 _(Check source code or version.php)_ | V **Search Databases** 📂 _(Check CVE records & Exploit DB)_ | V **Check CVSS Score** 🌡️ _(Is it an 8, 9, or 10? If yes, fix it immediately!)_ | V **Test the Exploit** 🧪 _(Use a "Proof of Concept" to see if the site is actually breakable)_ | V **GOAL: Remote Code Execution (RCE)** 🎯 _(Full control of the server)_

**Summary:** Always search for public exploits first. If the software is old or unpatched, there is likely already a "map" online telling you exactly how to break in. Use the CVSS score to prioritize what to fix first!
