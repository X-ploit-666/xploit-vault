When security experts test a website (called a **Penetration Test**), they look for weaknesses. These weaknesses can happen in two ways:

1. **Handmade Sites:** The code the developers wrote themselves might have bugs.
    
2. **Mistakes in Setup:** Even famous, public apps can be unsafe if the person setting them up makes a mistake (a **Misconfiguration**).
    

The examples below are part of the **OWASP Top 10**, which is a famous list of the ten most dangerous security risks for websites.

### 1. Broken Authentication and Access Control

These are very dangerous because they deal with who is allowed to enter the "locked doors" of a website.

- **Broken Authentication:** This is a bug that lets a hacker skip the login process. For example, a hacker might get into a site without a password, or a regular user might trick the site into thinking they are the "Admin" (the boss of the site).
    
- **Broken Access Control:** This happens when a hacker can get to pages they shouldn't see. Imagine a student being able to see the "Teacher’s Gradebook" page just by typing the right web address.
    

**Example:** In a program called _College Management System 1.2_, you can log in without an account just by typing a special code (`' or 0=0 #`) into the email box.

### 2. Malicious File Upload

Many websites let you upload photos or documents. If the site doesn't double-check what you are sending, a hacker can upload a **Malicious Script**.

- **What is a Script?** It is a file containing computer code (like **PHP**, a language used to build websites).
    
- **The Danger:** If a hacker uploads a PHP script instead of a photo, they can run commands on the website’s computer (the **Server**) and take control of it.
    

Even if developers try to stop this, hackers often find ways around the checks. **Example:** A tool for WordPress called _Responsive Thumbnail Slider_ lets hackers upload a virus by tricking the computer with two file types in one name (like `virus.php.jpg`).

### 3. Command Injection

Some websites run commands on their computer's main operating system (like Windows or Linux) to do tasks, like downloading a new plugin.

- **The Bug:** If the website doesn't clean the text a user types (**Sanitizing**), a hacker can "inject" or sneak in their own extra command.
    
- **The Result:** The computer runs the website's command _and_ the hacker's secret command at the same time, giving the hacker full control.
    

**Example:** A tool called _Plainview Activity Monitor_ lets hackers add a vertical bar (`|`) followed by a secret command to steal data.

### 4. SQL Injection (SQLi)

This is very similar to Command Injection, but it targets the **Database** (where the website stores all its information, like names and passwords).

- **SQL:** This is the special language used to talk to databases.
    
- **The Bug:** If a website takes what a user types and puts it directly into a database search without checking it, the hacker can change the search.
    

**Example:** In that same _College Management System_, a hacker can write a piece of SQL code that always says "True." This tricks the database into letting them log in or showing them everyone's private data.

### 💻 Code Breakdown

In the text, we see this line of PHP code used to search for users:

```
$query = "select * from users where name like '%$searchInput%'";
```

|Part of Code|What it means|
|---|---|
|`$query`|This is a "box" (variable) that holds the text command we want to send to the database.|
|`select *`|This tells the database: "Grab every single piece of information..."|
|`from users`|"...from the list (table) named 'users'."|
|`where name like`|"Only show me the people whose names match what I’m about to type."|
|`'%$searchInput%'`|This is the danger zone. `$searchInput` is exactly what the user typed. The `%` signs mean "find anything that contains this text."|

**Why is this dangerous?** If a user types a normal name like "Bob," the code works fine. But if a user types something like `' OR '1'='1`, they can trick the database into showing every user's info because `1=1` is always true. This is called **SQL Injection**.

### 📊 Quick Summary Map

**USER INPUT** ➡️ **THE WEBSITE** ➡️ **THE DANGER**

- **Login Box** ➡️ No Check ➡️ **Auth Bypass** (Entry without password)
    
- **Upload Button** ➡️ Bad Validation ➡️ **Web Shell** (Hacker runs code)
    
- **Search Bar** ➡️ Not Sanitized ➡️ **Command Injection** (Control the Server)
    
- **Database Search** ➡️ Direct Input ➡️ **SQL Injection** (Steal all Data)
    

_Understanding these basics is the first step toward becoming a security expert. You will see these same patterns everywhere in the world of cybersecurity!_
