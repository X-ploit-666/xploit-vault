### What is it?

Imagine every website is a large building. Usually, if you want to find a specific room, you have to look at a map or ask someone. **RFC 8615** is a rule that says every website "building" should have one specific "front desk" area. This area is always located at `/.well-known/`.

A **URI** (Uniform Resource Identifier) is just a fancy name for a web address or link. The **.well-known** path is a standard folder on a website's main server. It holds important **metadata** (which is just "data about other data"). This folder centralizes configuration files and details about how the website handles security, services, and technical rules.

### Why do we use it?

Because this folder is always in the same place, web browsers and apps don't have to guess where to find information. They can automatically build a link to get what they need. For example, if a tool wants to see a site's security rules, it knows to go straight to `https://example.com/.well-known/security.txt`.

The **IANA** (an organization that keeps track of internet numbers and names) maintains a **registry** (a master list) of all the official names allowed in this folder.

### 📋 Common Examples of .well-known Files

|Name|What it does|Status|
|---|---|---|
|**security.txt**|Tells security researchers who to contact if they find a bug or a safety hole.|Permanent|
|**change-password**|A standard link that takes a user directly to the page where they can update their password.|Provisional (Temporary)|
|**openid-configuration**|Details for **OpenID Connect** (a system that lets you log in using another account, like "Sign in with Google").|Permanent|
|**assetlinks.json**|Used to prove that a website and a mobile app are owned by the same company.|Permanent|
|**mta-sts.txt**|Sets security rules for emails to make sure they aren't intercepted.|Permanent|

### 🕵️ Web Recon (Information Gathering)

Security experts use these URIs during **Web Recon** (the process of researching a website to find weaknesses). By looking in the `.well-known` folder, they can find hidden settings.

A very important one is `openid-configuration`. It tells the expert exactly how the website handles logins.

#### 💻 The Configuration Code

When a program looks at `https://example.com/.well-known/openid-configuration`, it receives a **JSON** file (a text-based way to store data). Here is the code and what it means:

```
{
  "issuer": "[https://example.com](https://example.com)",
  "authorization_endpoint": "[https://example.com/oauth2/authorize](https://example.com/oauth2/authorize)",
  "token_endpoint": "[https://example.com/oauth2/token](https://example.com/oauth2/token)",
  "userinfo_endpoint": "[https://example.com/oauth2/userinfo](https://example.com/oauth2/userinfo)",
  "jwks_uri": "[https://example.com/oauth2/jwks](https://example.com/oauth2/jwks)",
  "response_types_supported": ["code", "token", "id_token"],
  "subject_types_supported": ["public"],
  "id_token_signing_alg_values_supported": ["RS256"],
  "scopes_supported": ["openid", "profile", "email"]
}
```

**What this code tells us:**

- **issuer**: The main "identity" of the server.
    
- **authorization_endpoint**: The specific link where users are sent to log in.
    
- **token_endpoint**: The "back-door" link where the app gets a digital "key" (token) to prove the user is logged in.
    
- **userinfo_endpoint**: The place to ask for the user's name or email.
    
- **jwks_uri**: This holds the **cryptographic keys** (digital locks) the server uses to sign its data.
    
- **response_types & scopes**: These tell us what the login system is capable of doing (like sharing your profile or email).
    
- **signing_alg**: The specific math formula (**algorithm**) used to keep the login keys secure.

### Conclusion

By checking these standard locations, security professionals can map out a website's "security landscape." It reveals exactly how a site is built and what security measures are being used.
