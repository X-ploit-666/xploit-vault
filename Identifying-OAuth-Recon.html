Mapping the
OAuth Surface

Before you exploit, you must identify. We look for the easy clues on the surface and the technical proof hidden in the traffic.

Level 1: Visual Detection

The most obvious indicator. If the UI presents federated identity, the target is using OAuth.

"Login with Google"
"Sign in with Facebook"

Level 2: Backend Confirmation

Intercepting the traffic via Burp Suite. Look for the initial GET /authorization request.

GET /o/oauth2/v2/auth? HTTP/1.1

client_id=target_app_id

redirect_uri=https://app.com/callback

response_type=code

scope=email profile

state=csrf_protection_token

📡

Advanced Recon: The Config Query

The true hacker doesn't stop at the UI. We probe for hidden endpoints that leak the system's "Security Manual."

/.well-known/openid-configuration

Returns JSON revealing endpoints, supported flows, and cryptographical features.

/.well-known/oauth-authorization-server

The secondary target for mapping internal OAuth logic.

01

The Clue

"Use company badge to enter" signs confirm external auth is in use.

02

Watching employees go to specific checkpoints (/authorization) to show ID.

03

The Manual

Finding the left-behind security manual (JSON config) revealing all entry rules.

Execution Example: Google Recon

// Intercepted initial request

GET https://accounts.google.com/o/oauth2/v2/auth?client_id=...

Result: OAuth Detected.

// Probing for config

GET https://accounts.google.com/.well-known/openid-configuration

{

"issuer": "https://accounts.google.com",

"authorization_endpoint": ".../o/oauth2/v2/auth",

"token_endpoint": ".../oauth2/v4/token",

"userinfo_endpoint": ".../oauth2/v3/userinfo"

}
