## Mappings To Ohter Lists ( CAPEC )

![[Capec.png]]

After months of research, the experts created a list of **19 specific ways** bots abuse websites. They gave each one a unique code (like OAT-001).

The goal wasn't to describe _how_ a hacker feels or _why_ they do it (the outcome), but rather **"What is the bot doing to the website right now?"** For example, instead of calling something "Malware Distribution," they might call the action **Account Creation** (OAT-019) because that is the actual step being automated. They also made sure these names weren't tied to any specific brand or technology—they work for any website.

## 🔑 Key Concepts

- **Ontology:** The formal list and naming system for these 19 threats. 🏷️
    
- **OAT-XXX:** The identification code for each threat (OWASP Automated Threat).
    
- **Asset:** What the bot is targeting (e.g., user data, gift card balances, or server CPU). 💎
    
- **Abuse of Functionality:** Using a "normal" feature (like a login or a search) in a way that hurts the business.
    
- **Divergence:** When the application starts doing something it wasn't intended to do because of the automation.
    

## ⚠️ Threat Explanation

An **Automated Threat** occurs when a "Threat Agent" (a bot or a person using a script) interacts with a "Web Asset" (a page or API). This interaction is **highly iterative** (happens over and over) and **multi-step**. The threat isn't a single "exploit" but a pattern of behavior that wears down the site or steals data bit by bit.

## 🛠 Methodology

To use this ontology like a pro, follow these steps:

1. **Identify the Action:** Don't look at the result yet. Look at the _action_. Is the bot clicking a button? Filling a form? Requesting a file?
    
2. **Consult the OAT List:** Match the action to one of the 19 codes (e.g., if it's guessing passwords, it's **OAT-007 Credential Cracking**).
    
3. **Check the Subset:** Use the handbook's categories to see if it relates to **Credentials**, **Payments**, or **Vulnerability Discovery**.
    
4. **Cross-Reference:** Check if this OAT matches a known **CAPEC** (Attack Pattern) or **WASC** (Classification) ID to see how others describe it.
    

## 🎯 Real Bug Bounty Context

When reporting a bug, using an OAT code makes you look like an expert.

- **Example:** Instead of saying "I can spam your register page," you say, "**OAT-019 Account Creation** is possible due to a lack of anti-automation on the `/register` endpoint."
    
- **Focus:** Bug bounty programs care about the "Magnitude of Impact." While the OAT list doesn't rank them, you should explain how the OAT you found affects the "Parties Affected" (e.g., "This hurts your genuine users by slowing down the site").
    

## 🖼 Visual Explanation

The handbook organizes these 19 threats into logical "buckets" or subsets:

|---|---|---|
|**Account/Identity** 👤|Stealing or making accounts.|OAT-019 (Creation), OAT-008 (Stuffing)|
|**Payments** 💳|Stealing money or card info.|OAT-001 (Carding), OAT-012 (Cashing Out)|
|**Discovery** 🔍|Finding out how the site works.|OAT-004 (Fingerprinting), OAT-014 (Scanning)|

## 🏗 Example Scenario

You are testing a site that sells limited-edition posters.

- **The Action:** You notice a script is hitting the "Add to Cart" button 500 times a second for a specific item.
    
- **The Ontology Match:** This is **OAT-005 Scalping**. The bot is trying to get limited goods before humans can.
    
- **The "Cheating" Factor:** Even if it's not "illegal" in all countries, the handbook notes this is "Cheating" and undermines user trust.
    

## 💡 Important Notes

- **Generic Names:** Some names are "buckets." **OAT-011 Scraping** covers a lot of different types of data harvesting.
    
- **The "OAT" Prefix:** In your head, always put "Automated Web Application" before the name. (e.g., Automated Web Application Denial of Service).
    
- **Discarded Terms:** Notice that **Man-in-the-Browser** was excluded because it happens on the _user's_ computer, not the web app itself. 💻❌
    

## 🔄 Quick Review

The Ontology provides **19 unique OAT codes** that focus on the **action** being performed by a bot. These are categorized into subsets like **Credentials** and **Payments** to help you identify exactly what kind of "Abuse of Functionality" is happening.
