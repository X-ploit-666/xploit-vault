### 0x01: SYSTEM PROTOCOL (THE GOAL)
In the OAuth framework, the `redirect_uri` acts as the GPS for the authentication result.
*   **The Payload:** Authorization Code or Access Token.
*   **The Security Layer:** A strict **Whitelist** ("Only send data to these trusted URLs").

---

### 0x02: THE GLITCH (WHERE IT BREAKS)
Vulnerabilities emerge when the server performs **Loose Validation** instead of **Strict Matching**. 

**Primary Failure Points:**
*   **Prefix Checking:** Only verifying the start of the URL string.
*   **Path Permissiveness:** Allowing unauthorized extra characters or sub-paths.
*   **Parser Discordance:** Misinterpreting complex URL formats or encoding.

---

### 0x03: PAYLOAD VECTORS (BYPASS ARSENAL)
Even when `evil.com` is explicitly blocked, the validation logic can be tricked using these specific vectors:

*   **Subdomain Hijack:** `https://trusted.com.evil.com`
*   **Path Traversal:** `https://trusted.com/../evil`
*   **Parameter Chaining:** `https://trusted.com?redirect=evil.com`
*   **Parameter Pollution:** `redirect_uri=trusted.com&redirect_uri=evil.com`
*   **URL Obfuscation:** `trusted.com&@evil.com`
*   **Localhost Spoofing:** `localhost.evil.com`

> [!NOTE]
> Changing the `response_mode` parameter can also shift how the browser interprets these URLs, often bypassing server-side filters.

---

### 0x04: THE ATTACK CHAIN
1.  **Preparation:** Attacker crafts an OAuth request with a manipulated `redirect_uri`.
2.  **Execution:** The victim logs in via the legitimate OAuth provider.
3.  **Exfiltration:** The OAuth server passes the validation (thinking it’s trusted) and sends the **Code/Token** to the attacker-controlled endpoint.
4.  **Takeover:** The attacker uses the stolen data to impersonate the victim or call protected APIs.

---

### 0x05: ANALOGY (THE NEGLIGENT GUARD)
Think of a security guard checking delivery addresses.
*   **The Rule:** "Only allow deliveries to **trusted.com**."
*   **The Bad Check:** The guard only glances at the beginning of the label.
*   **The Trick:** An attacker writes **"trusted.com &rarr; forward to attacker's house"** on the label.
*   **The Result:** The guard sees "trusted.com," waves it through, and your package is stolen.

---

### 0x06: TECHNICAL ROOT CAUSE
The vulnerability exists because the server prioritizes **Pattern Matching** (loose/confusing) over **Identity Verification** (strict/exact).

**Vulnerable Validation:**
`https://client-app.com/callback?next=evil.com` (Allowed due to path leakage)
`https://client-app.com.evil.com` (Allowed due to prefix-only checking)

---

### 0x07: FINAL LOG (ONE-LINE TAKEAWAY)
If `redirect_uri` validation is not **strict and exact**, attackers can leverage URL parsing tricks to exfiltrate OAuth secrets and achieve full account takeover.
