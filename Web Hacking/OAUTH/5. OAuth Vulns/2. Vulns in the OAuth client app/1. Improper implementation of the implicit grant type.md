### Feynman-style explanation (simple)

In the **implicit flow**, the login works like this:

1. You log in via an external provider
    
2. The provider sends back an **access token in the browser (URL)**
    
3. The website’s JavaScript reads that token
    
4. Then it sends a request to its own server like:
    
    ```
    POST /authenticate
    email=your@email.com
    token=XYZ
    ```
    
5. The server trusts this and logs you in
    

---

### Where the problem happens

The server:

- **does not have a password**
    
- **does not verify deeply**
    
- just trusts:
    
    - email
        
    - token
        

👉 But this request is sent from the browser → **you can modify it**

If the server **doesn’t check that the token actually belongs to that email**, then:

> You can send someone else’s email with your token

and the server will say:

> “Looks fine → logged in”

---

### Why this is dangerous

Because:

- everything happens in the browser
    
- the request is visible and editable
    
- the server blindly trusts user-controlled data
    

👉 This turns login into **“trust whatever the client says”**

---

### Analogy

Think of a club entry system:

- You show a **ticket (token)** and say your **name (email)**
    
- The guard does NOT check if the ticket matches your name
    
- He just checks:
    
    > “You have a ticket? Okay, go in as whoever you claim”
    

So you can say:

- “I’m VIP John”
    
- show your own ticket
    

👉 and get in as John

### Real-world example (your lab)

You captured:

```
POST /authenticate
email=wiener@...
token=XYZ
```

Then you changed:

```
email=carlos@...
token=XYZ
```

Server response:  
→ no error  
→ login successful

👉 Because the server:

- did NOT verify token ↔ user binding
    
- trusted client input
    

---

### Root issue (technical)

- Token is obtained in browser (implicit flow)
    
- Client sends identity + token to backend
    
- Backend fails to:
    
    - validate token properly
        
    - check token belongs to that user
        

---

### One-line takeaway

Implicit flow becomes dangerous when the server **trusts user-supplied identity data without verifying it against the access token**, allowing **account impersonation**.
