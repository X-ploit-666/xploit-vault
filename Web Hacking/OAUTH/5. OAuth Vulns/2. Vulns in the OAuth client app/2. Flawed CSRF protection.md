### Feynman-style explanation (simple)

When a website uses OAuth, there’s a small but important value called **`state`**.

👉 Its job is simple:

> Make sure the login process that _finishes_ is the same one the user _started_

---

### Why this matters

Without `state`, the website can’t tell:

- “Did **this user** start this login?”
    
- or
    
- “Was this login started by someone else and forced onto the user?”
    

---

### What can go wrong

If `state` is missing:

1. Attacker starts an OAuth flow **on their own account**
    
2. Gets a valid **authorization code**
    
3. Sends a link to the victim’s browser (or loads it invisibly)
    
4. Victim’s browser completes the flow
    

👉 The website thinks:

> “This user just linked/logged in normally”

But actually:

> The victim just got linked to the **attacker’s account**

---

### What `state` should do

- It’s a **random, unguessable value**
    
- Tied to the user’s session
    
- Sent in request → returned in response
    

👉 If it doesn’t match → request is rejected

Without it:

> No protection against CSRF-style attacks

---

### Analogy

Think of ordering food at a restaurant:

- You place an order → get a **ticket number (state)**
    
- When food is ready → they call your number
    

Now imagine there’s **no ticket system**:

- Anyone can bring a plate and say:
    
    > “This is your order”
    

👉 You might accept **someone else’s food** without realizing

### Real-world example (your lab)

Flow:

1. You start “Attach social profile”
    
2. You intercept:
    
    ```id="r7k4zt"
    GET /oauth-linking?code=ATTACKER_CODE
    ```
    
3. You **don’t complete it** (so code stays valid)
    
4. You embed it in:
    
    ```html
    <iframe src="https://victim-site.com/oauth-linking?code=ATTACKER_CODE"></iframe>
    ```
    
5. Victim loads this page
    

👉 Their browser completes the request

---

### Result

- Victim’s account gets linked to **attacker’s social account**
    
- Now attacker logs in via OAuth
    
- Gets access to victim (even admin)
    

---

### Root issue (technical)

- No `state` parameter
    
- No way to bind request ↔ user session
    
- Server accepts unsolicited OAuth responses
    

---

### One-line takeaway

Missing `state` means the app can’t verify **who started the OAuth flow**, enabling attackers to **force actions (like account linking) on victims via CSRF**.
