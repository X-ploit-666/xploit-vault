###  Feynman-style explanation (simple)

How do you _know_ a website is using OAuth?

There are two levels:

---

### 1. Easy clue (what you see)

If a site shows buttons like:

- “Login with Google”
    
- “Login with Facebook”
    

👉 That’s a strong sign it’s using OAuth

---

### 2. Real confirmation (what happens behind the scenes)

When you inspect the traffic (for example with Burp Suite), you’ll always see a special request at the start:

```
GET /authorization?...parameters...
```

This request goes to the OAuth provider and contains key fields:

- `client_id` → identifies the app
    
- `redirect_uri` → where to send the user back
    
- `response_type` → which flow (code or token)
    
- `scope` → what data is requested
    
- `state` → protects against request tampering
    

👉 If you see this request, it’s definitely OAuth

---

### What is “recon” here?

Once you know OAuth is used, you gather info about the OAuth service.

You look at:

- where the request is sent (the provider)
    
- how endpoints behave
    
- what features are enabled
    

---

### Hidden useful endpoints

You can often query:

```
/.well-known/oauth-authorization-server
/.well-known/openid-configuration
```

These return a JSON config with:

- endpoints (`/authorize`, `/token`, `/userinfo`)
    
- supported flows
    
- extra features
    

👉 This helps map the attack surface

---

## Analogy

Think of investigating a **security checkpoint system**:

### Step 1 (visible clue):

You see signs like:

> “Use your company badge to enter”

→ You know external authentication is used

---

### Step 2 (behind the scenes):

You watch how people enter:

- They go to a checkpoint (`/authorization`)
    
- Show ID
    
- Get redirected back
    

→ Now you understand the system flow

---

### Step 3 (recon):

You find the **security manual**:

- rules
    
- entry points
    
- allowed badge types
    

→ This reveals weaknesses or extra paths

---

## Real-world example

You test a website with “Login with Google”:

1. Click login
    
2. In Burp Suite, you see:
    
    ```
    GET https://accounts.google.com/o/oauth2/v2/auth?client_id=...
    ```
    
3. You confirm OAuth is used
    

Then you try:

```
https://accounts.google.com/.well-known/openid-configuration
```

You get:

- token endpoint
    
- userinfo endpoint
    
- supported scopes
    

👉 Now you understand:

- how login works
    
- where to probe for bugs
    

---

### One-line takeaway

To identify OAuth:

- **Frontend clue:** “Login with X”
    
- **Backend proof:** `/authorization` request with OAuth parameters
    
- **Recon:** query `.well-known` endpoints to map the system
